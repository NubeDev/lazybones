The lazybones MCP service exposes the daemon over the Model Context Protocol so an agent (Claude Desktop, the claude CLI, Cursor, a custom rmcp client) can drive lazybones through typed tools instead of hand-rolled curl.

\n

It is mounted in-process inside lazybonesd at POST /mcp over the streamable-HTTP transport. There is no separate process and no second store — the tools call the same StoreHandle the REST routes use, so the two surfaces can never drift in behaviour.

\n

Verified live on 2026-06-25: the initialize handshake reports lazybones-mcp v0.1.0, and tools/list returns 58 tools across four groups — orchestration, documents, extensions, and supervision.

Authentication mirrors REST exactly: a bearer token in the Authorization header resolves to a ScopedSession, and every tool re-checks the session capability before acting.

\n
    \n
  • Read tools (state / logs / history, and all *.list / *.get) need no token.
    • \n
  • Authoring tools need Author (tasks, workflows, templates, skills) or Document (documents, branding, assets).
    • \n
  • Lifecycle tools (workflow start/stop, task retry/cancel) need Block / Claim, which the default Author profile lacks — they return 403.
    • \n
  • Installing extensions and reading secrets are loop-only and never agent-reachable.
    • \n
\n

The one rule: authoring is not running. An MCP client may freely create work; starting it stays gated behind a capability the default token does not hold.
